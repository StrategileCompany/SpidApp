# Normas Corporativas ClusterOS

**Versão:** 1.0
**Data de emissão:** 2026-04-16
**Aplica-se a:** todos os projetos do ecossistema ClusterOS (StrategileCompany + GeraldoPFJr)

## 1. Identidade e escopo

ClusterOS é a **entidade corporativa** que rege a operação dos projetos de software do Geraldo. Cada projeto é uma subsidiária — tem stack própria, regras específicas, mas segue as normas deste documento.

Quando um agente (Claude Code, Codex, Jarvis, Gemini CLI) opera em qualquer projeto do ecossistema, **deve seguir estas normas antes das regras específicas do projeto**. Se houver conflito entre norma corporativa e regra de projeto, a norma corporativa prevalece — a menos que o CLAUDE.md do projeto marque explicitamente `## Excecao Aprovada` com justificativa.

**Ecossistema atual** (ao emitir v1.0): 30 projetos em `~/projects/`, 14 deles submodules de ClusterOS. Repositórios reais em StrategileCompany/ClusterOS, StrategileCompany/*, GeraldoPFJr/*.

## 3. Formato do BACKLOG.md (norma central)

Todo BACKLOG.md deve seguir exatamente este formato para ser processado pelos agentes automatizados (backlog-formatter, Jarvis `read_backlog`, Intelligence analyzer).

### 3.1. Cabeçalho

```markdown
# <Nome do Projeto> — Backlog Completo
Ultima Revisao: YYYY-MM-DD
Ultimo Commit: <hash-curto> (<mensagem>)
Sprint Ativo: <descricao curta do sprint atual>
Proximo ID: T-NNN
```

- `Ultima Revisao`: data ISO da última edição significativa.
- `Ultimo Commit`: opcional, útil para rastreio.
- `Sprint Ativo`: frase curta descrevendo o foco atual.
- `Proximo ID`: o próximo número T-NNN disponível. **Nunca reutilizar IDs**.

### 3.2. Seções obrigatórias

```markdown
---

## Tasks Ativas

### T-NNN [tipo] [modulo] Titulo curto
- **Status:** pendente | em andamento | concluido | bloqueado | cancelado
- **Criado:** YYYY-MM-DD
- **Concluido:** YYYY-MM-DD — <hash-do-commit>   (apenas se status = concluido)
- **Descricao:** texto livre, pode ter várias linhas, incluir subtarefas, critérios de aceite

---

## Historico
(tasks concluidas, canceladas ou movidas daqui — nunca deletar)
```

### 3.3. Regras das tasks

1. **ID permanente**: `T-NNN` é único e imutável. Nunca reciclar. `TL-NNN` reservado para tickets legacy importados de outros sistemas.
2. **Tipo obrigatório entre colchetes**: `feat`, `fix`, `chore`, `docs`, `refactor`, `test`, `ops`, `security`, `infra`.
3. **Módulo opcional**: identifica parte do sistema (`cluster-web`, `backend`, `api`, `flutter-app`, etc).
4. **Status controlado**: apenas os 5 valores listados.
5. **Ao concluir uma task**:
   - Trocar `**Status:** pendente` para `**Status:** concluido`
   - Adicionar linha `**Concluido:** YYYY-MM-DD — <hash>` com hash do commit que entrega a task
   - **Mover a task inteira** de `## Tasks Ativas` para `## Historico`
   - **Nunca deletar**. Histórico é permanente.
6. **Ao criar nova task**:
   - Usar o número em `Proximo ID` do cabeçalho
   - Incrementar `Proximo ID` no cabeçalho (se `Proximo ID: T-042`, use T-042 e atualize para T-043)
   - Adicionar na seção `## Tasks Ativas` com status `pendente`

### 3.4. Exemplo completo


## 5. Regra de propagação ao concluir task

Ao concluir qualquer task (`**Status:** concluido`), o agente **deve** propagar o resultado para os documentos afetados. Regra dos 4 arquivos:

| Mudou... | Atualizar em... |
|---|---|
| Feature nova entregue | `PRODUTO.md`: marcar `- [x]` na feature, `BACKLOG.md`: mover task para Histórico |
| Rota/módulo/infra nova | `TECNICO.md`: adicionar na tabela de rotas/módulos, `BACKLOG.md`: Histórico |
| Bug corrigido | `BACKLOG.md`: Histórico (sem tocar outros) |
| Refactor/chore | `TECNICO.md` se mudou arquitetura, senão só `BACKLOG.md` |
| Info factual obsoleta no `CLAUDE.md` | Atualizar o `CLAUDE.md` do projeto |

**Máximo 4 arquivos** por task. Se a task tocou nada no código (ex: discussão), não atualizar doc nenhum.

## 6. Chain de providers LLM (Classe A)

Qualquer projeto do ecossistema que faça chamadas a LLM em componentes de **alta importância** (assistentes conversacionais, agentes de ação, crons analíticos) deve seguir esta chain de fallback:

1. **Principal**: Gemini 2.5 Flash (Google Cloud, projeto `cluster-gemini`)
2. **Fallback 1**: OpenAI `gpt-4o-mini`
3. **Fallback 2**: OpenRouter (`google/gemini-2.0-flash-001` ou equivalente)

Implementação de referência: `cluster-worker/src/provider-adapter.ts` (Node.js). Espelho no cluster-web: `cluster-web/lib/llm/client.ts`.

Critério de fallback: erros transientes (429, 503, 502, 500, UNAVAILABLE, timeout) disparam próximo provider. Erros de autenticação/quota absoluta não — abortam com mensagem clara.

**Tracing**: toda chamada Classe A deve passar pelo Langfuse (`LANGFUSE_HOST`, `LANGFUSE_PUBLIC_KEY`, `LANGFUSE_SECRET_KEY`). Init lazy — não bloquear se não configurado.

Componentes de **baixa importância** (ex: geração de rascunhos pontuais) podem usar qualquer provider direto.

## 7. Segurança e secrets

### 7.1. Denylist de commit

Arquivos que **nunca** devem ser commitados em nenhum projeto:

- `.env`, `.env.local`, `.env.production`, `.env.*` (qualquer variante)
- `*.key`, `*.pem`, `*.pfx`, `*.p12`
- `credentials.json`, `credentials*.json`
- `service-account*.json`
- Qualquer arquivo contendo strings começando com `AKIA`, `AQ.`, `sk-`, `ghp_`, `github_pat_`

Projetos devem ter essas entradas no `.gitignore`. ClusterOS tem git hook global que bloqueia pushes com esses padrões.

### 7.2. Tokens em código

- **Jamais** hardcodear. Sempre via variável de ambiente.
- Nomes de env vars em UPPER_SNAKE_CASE.
- Documentar env vars obrigatórias no `CLAUDE.md` do projeto.

### 7.3. Rotação

- Se um secret for commitado acidentalmente: (1) revogar imediatamente no provider, (2) remover do histórico via BFG ou git-filter-repo, (3) force-push após aprovação do usuário, (4) registrar em `DEPENDS-ON-USER.md` do ClusterOS.

## 8. Nomenclatura de branches

- `main` ou `master` — branch default (cada projeto mantém sua convenção histórica, não forçar migração).
- `feature/<descricao-curta>` — features manuais.
- `fix/<descricao-curta>` — correções manuais.
- `agent/{tipo}/{ticket-id}` — branches criadas por agent worker (automáticas).
  - Exemplo: `agent/bug/T-273`, `agent/feature/T-274`.

## 9. Fluxo de agentes no ecossistema

Cada tipo de agente tem papel definido:

| Agente | Onde roda | O que faz |
|---|---|---|
| **Claude Code (Max)** | local ou VPS via tmux `claude-*` | coding sob demanda, tickets manuais |
| **Codex (GPT Plus)** | VPS via cron | tarefas agendadas (briefings, reviews, scans) |
| **Jarvis** | cluster-worker na VPS | assistente Telegram, 19 tools |
| **Intelligence Analyzer** | cluster-web na VPS | cron 08:00 UTC, escreve em `~/brain/wiki/analyses/` |
| **Backlog Formatter** | cron VPS 03:30 UTC | reformata BACKLOGs fora do padrão |
| **Standards Propagator** | cron VPS 03:45 UTC | propaga este STANDARDS.md aos projetos |

**Regra de ouro**: Claude Code Max **nunca** roda em rotinas automáticas — preserva quota. Rotinas usam Codex, Gemini CLI ou scripts puros.

## 11. Invariantes (nunca negociáveis)

Estas 5 regras são absolutas. Qualquer agente que as viole deve ser corrigido:

1. **IDs de task são permanentes** — nunca reciclar T-NNN.
2. **Histórico nunca é deletado** — tasks concluídas vão para `## Historico`, permanecem para sempre.
3. **Secrets jamais no git** — denylist em §7.1 é bloqueante.
4. **Máximo 4 arquivos propagados por task** — evitar churn.
5. **Este STANDARDS.md é auto-gerado nos projetos** — editar apenas o master em ClusterOS/.docs/.
